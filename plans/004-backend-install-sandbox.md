# Backend Install Sandbox

This ExecPlan is a living document. The sections Progress, Surprises & Discoveries, Decision Log, and Outcomes & Retrospective must be kept up to date as work proceeds. This plan follows the format described in .agent/PLANS.md.

## Why this exists

Phase 4 needs a local way to check whether optional backend dependencies can be installed without touching the developer's active Python environment. The user-visible result is a repeatable command that creates isolated environments under `sandbox/`, captures install logs, and writes a concise summary for the `core`, `mineru`, and `paddleocr_vl` backend dependency sets.

## Scope

In scope:
- Add `scripts/check_backend_installs.sh`.
- Keep virtual environments, pip caches, logs, and reports under `sandbox/backend-installs/`.
- Install the requested dependency sets for `core`, `mineru`, and `paddleocr_vl` using Python `venv`.
- Run dependency import smoke tests without downloading models.
- Write one backend log and one aggregate Markdown summary.
- Add documentation for the command.
- Add a lightweight pytest check for the script and sandbox ignore rule.

Out of scope:
- Backend extraction quality validation.
- Model downloads or OCR/layout benchmark execution.
- Adding optional backend packages to `requirements.txt`.
- Introducing runtime ML dependencies into the default package.

## Target behavior

After this work, a contributor can run:

    cd /home/jgarcia/pdf2md/pdf2md
    bash scripts/check_backend_installs.sh core
    bash scripts/check_backend_installs.sh mineru
    bash scripts/check_backend_installs.sh paddleocr_vl

Each command creates or refreshes only the selected backend's environment under `sandbox/backend-installs/<backend>/`, writes the backend log to `sandbox/backend-installs/logs/<backend>.log`, and rewrites `sandbox/backend-installs/summary.md` with rows for the selected backend runs. The summary records the Python version, pip version, install result, likely failed package, last relevant error lines, and import smoke-test result.

## Design and decisions

The script will be venv-first because the requested `envs/*.yml` files are absent in this checkout and a venv fallback is mandatory. It will use the interpreter that runs the script by default, with `PYTHON_BIN` available as an override for local experimentation. It will never activate an environment; every command will call the venv's `python` executable directly.

The generated state root is `sandbox/backend-installs`. The script will set `PIP_CACHE_DIR` to `sandbox/backend-installs/pip-cache` and `PYTHONPYCACHEPREFIX` to `sandbox/backend-installs/pycache` so dependency caches and bytecode generated by the script stay under `sandbox/`.

For each backend, the install set is:
- `core`: `requirements.txt` and `pytest`.
- `mineru`: `requirements.txt`, `pytest`, and `mineru`.
- `paddleocr_vl`: `requirements.txt`, `pytest`, `paddleocr`, and `paddlepaddle`.

Smoke tests are imports only. `core` must import `fitz`, `yaml`, and `doc2md`. `mineru` must import `mineru`; it also attempts `magic_pdf` and records whether it is available. `paddleocr_vl` must import `paddleocr` and `paddle`.

## Risks and rollback notes

The heaviest risk is installing optional packages such as `paddlepaddle`, which can take time, require compatible wheels, or fail behind a blocked network. Those failures are expected validation outcomes as long as they are logged and summarized. The script is safe to rerun: it removes only `sandbox/backend-installs/<backend>/venv` for the selected backend and then recreates it.

Rollback is straightforward: remove `scripts/check_backend_installs.sh`, `docs/backends.md`, `tests/test_backend_install_sandbox.py`, and this plan file, then remove the `sandbox/` line from `.gitignore` if no longer desired.
