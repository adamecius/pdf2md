# Plan 0003: Backend Orchestration and Visual Models

## Goal

Create a flexible backend architecture that can use deterministic extraction, Docling, MinerU, PaddleOCR-VL, GLM-OCR, dots.ocr, FireRed-OCR, Marker and other offline models.

The goal is not to choose one winner. The goal is to orchestrate several backends robustly.

## Backend interface

Create:

```text
doc2md/backends/base.py
doc2md/backends/registry.py
```

Base concepts:

```python
class ExtractionBackend:
    name: str

    def available(self) -> bool:
        ...

    def extract_document(self, input_path, options) -> DocumentIR:
        ...

    def extract_page(self, page_image, page_context, options) -> PageIR:
        ...
```

Backends may support document-level extraction, page-level extraction or both.

## Backend registry

The registry should allow:

```bash
python -m doc2md input.pdf --backend deterministic
python -m doc2md input.pdf --backend docling
python -m doc2md input.pdf --backend mineru
python -m doc2md input.pdf --backend paddleocr_vl
python -m doc2md input.pdf --backend auto
```

Auto mode should produce a backend plan, not a single arbitrary choice.

## Backend candidates

Implement backend stubs with lazy imports:

```text
deterministic.py
docling_backend.py
mineru_backend.py
paddleocr_vl_backend.py
glm_ocr_backend.py
dots_ocr_backend.py
firered_backend.py
marker_backend.py
olmocr_backend.py
```

Every backend module must:

- import heavy dependencies lazily
- return clear install messages
- preserve raw outputs
- convert to DocIR
- not break default tests if not installed

## Backend priority for offline robust processing

### 1. Deterministic PyMuPDF

Always available. Useful for:

- text-layer extraction
- page profiling
- rendering page images
- detecting bad encodings
- providing baseline text

### 2. Docling

Useful for:

- structured document representation
- book-like hierarchy
- tables
- chunking ideas
- comparison against a mature architecture

### 3. MinerU

Useful for:

- strong all-in-one document parsing
- scientific document layouts
- formulas, tables and figures
- high-quality offline parsing

Since licence is not a blocker, MinerU can be treated as a high-priority backend.

### 4. PaddleOCR-VL-1.5

Useful for:

- compact visual model parsing
- OCR
- formulas
- tables
- scanned and distorted pages

### 5. GLM-OCR

Useful for:

- strong visual OCR
- document layout pipeline
- high quality specialised parsing

### 6. dots.ocr

Useful for:

- structured visual output
- bounding boxes
- Markdown, LaTeX and HTML-like outputs
- DocIR-friendly layout extraction

### 7. FireRed-OCR

Useful for:

- additional strong visual candidate
- robustness through backend diversity

### 8. Marker and olmOCR

Useful for:

- alternative baselines
- book-scale comparison
- Markdown and linearisation comparison

## Orchestrator

Create:

```text
doc2md/orchestration/planner.py
doc2md/orchestration/executor.py
doc2md/orchestration/merger.py
doc2md/orchestration/cache.py
```

### Planner

Input:

- document profile
- page profiles
- user backend choice
- config
- available backends

Output:

- backend execution plan

Example:

```json
{
  "document_backend": "deterministic",
  "pages": [
    {
      "page_index": 0,
      "strategy": "visual",
      "backends": ["paddleocr_vl", "glm_ocr"],
      "reason": "cover page with large image regions"
    }
  ]
}
```

### Executor

Responsibilities:

- render pages
- batch page images
- run backends
- handle timeouts
- handle exceptions
- cache outputs
- record run metadata

### Merger

Responsibilities:

- combine page results
- select text candidates
- select formula candidates
- select table candidates
- link captions
- deduplicate headers and footers
- resolve reading order
- produce final DocIR

## Local GPU strategy

Support these execution modes:

```text
direct Python backend
local CLI backend
local HTTP endpoint
vLLM/SGLang-style endpoint
precomputed raw output folder
```

This lets the project use large models offline without forcing one integration style.

## Robustness features

Implement early:

- timeouts
- per-page failure handling
- raw output preservation
- cache keys based on input file hash and backend options
- retry policy
- page rendering reuse
- model output validation
- fallback to deterministic text
- backend availability checks
- manifest for every run

## Configuration example

```yaml
backend:
  mode: auto
  preferred:
    - deterministic
    - docling
    - mineru
    - paddleocr_vl
    - glm_ocr
    - dots_ocr

visual:
  dpi: 200
  batch_size: 4
  device: cuda
  timeout_seconds: 300

routing:
  force_visual_for:
    - scanned
    - bad_encoding
    - formula_dense
    - table_dense
    - diagram_dense

cache:
  enabled: true
  path: .cache/doc2md
```

## Acceptance criteria

- backends can be listed
- missing heavy backends do not break default tests
- deterministic backend works without visual dependencies
- visual backend stubs return clear messages
- auto mode produces a readable execution plan
- backend outputs are preserved before merging
