# Plan 0001: Robust Offline Document Parsing Architecture

## Goal

Upgrade `doc2md` into a robust, offline, backend-agnostic document parsing framework suitable for scientific books, technical PDFs, scanned documents and visual-model workflows.

The architecture should be inspired by:

- OmniDocBench style evaluation: typed blocks, formulas, tables, reading order and layout
- Docling style representation: structured document object, provenance, hierarchy, chunking and exporters
- visual document model pipelines: page images, crops, local GPU inference and model-specific adapters

The project should optimise for robustness and flexibility over dependency minimalism, while keeping the core install testable and lightweight.

## Product assumptions

- The product is not commercial.
- It will run offline.
- Local computers are strong enough for visual models.
- Licence restrictions are not the principal design constraint.
- Reproducibility, robustness and extensibility matter more than matching one specific public leaderboard.
- Heavy models are allowed, but must not be mandatory for the default test suite.

## Non-goals

- Do not make Markdown the canonical representation.
- Do not hard-code one backend as the only path.
- Do not run OmniDocBench by default in pytest.
- Do not commit external benchmark data.
- Do not require GPUs for unit tests.
- Do not require visual models for core installation.
- Do not implement full official OmniDocBench scoring before the internal DocIR and backend interface exist.

## Target architecture

```text
PDF or image input
  -> media renderer and cache
  -> profiler
  -> router
  -> backend planner
  -> backend executor
  -> candidate store
  -> merger and reconciler
  -> DocIR JSON
  -> exporters
       - Markdown
       - MyST Markdown
       - HTML debug
       - JSONL chunks
       - benchmark prediction formats
  -> benchmark runner
```

## Architectural decision 1: DocIR is canonical

Canonical output:

```text
document.docir.json
```

Derived outputs:

```text
document.md
document.myst.md
document.html
document.blocks.jsonl
document.benchmark.json
media/
```

All backends must output DocIR or be adapted into DocIR.

## Architectural decision 2: Preserve candidates

A strong visual model may produce better layout and equations, while deterministic extraction may produce better text for clean pages. The system must not discard either too early.

Introduce candidate preservation:

```text
backend output
 -> candidate DocIR or candidate PageIR
 -> candidate store
 -> merger
 -> final DocIR
```

This supports robustness and later benchmarking.

## Architectural decision 3: Use page images as first-class evidence

Scientific books and exams often need diagrams, equations, plots and tables. Text extraction alone is insufficient.

The media layer must support:

```text
media/page_0001.png
media/page_0002.png
media/crops/<block_id>.png
media/manifest.json
```

DocIR blocks must be able to reference media.

## Architectural decision 4: Benchmarking is separate from testing

Default unit tests validate code behaviour.

Benchmarks evaluate extraction quality.

Full OmniDocBench is an external benchmark and should be explicitly invoked.

## Architectural decision 5: Backends are optional but first-class

Visual and heavy backends are allowed and encouraged, but they must be isolated behind adapters.

The core package must still be testable without them.

## Initial implementation phases

### Phase 1: DocIR schema and serialisation

Implement `doc2md/ir`.

### Phase 2: deterministic backend emits DocIR

Refactor the existing PyMuPDF/profiler/router path so it emits minimal DocIR.

### Phase 3: exporters

Generate Markdown, HTML and JSONL chunks from DocIR.

### Phase 4: backend interface and registry

Add backend plugin interface.

### Phase 5: optional visual backend stubs

Add lazy-import stubs for Docling, MinerU, PaddleOCR-VL, GLM-OCR, dots.ocr, FireRed-OCR, Marker and olmOCR.

### Phase 6: OmniDocBench adapter

Add optional downloader and adapter.

### Phase 7: local model execution

Add GPU-friendly batch execution and caching for visual backends.

## Acceptance criteria

- `pytest -q` runs without network, GPU or external datasets.
- `python -m doc2md input.pdf --backend deterministic --output out` produces `document.docir.json`.
- Markdown is generated from DocIR, not directly from raw extraction.
- JSONL chunks contain source block IDs, page references and media references.
- Optional backend modules fail gracefully if dependencies are missing.
- Tiny OmniDocBench fixture tests pass.
- Full OmniDocBench remains optional and disposable.
