# Plan 0004: Benchmarking and Test Strategy

## Goal

Use OmniDocBench logic and similar benchmarks to improve robustness, while keeping unit tests fast and offline.

The project should distinguish:

```text
unit tests
local regression tests
optional external benchmarks
full official benchmark evaluation
```

## Principle

A benchmark is not a normal unit test.

Unit tests answer:

```text
Is the code correct?
```

Benchmarks answer:

```text
How good is the extraction?
```

Both are needed, but they must not be mixed.

## Unit tests

Default command:

```bash
pytest -q
```

Must run without:

- network
- GPU
- external datasets
- OmniDocBench
- visual model packages

Test with tiny fixtures:

```text
tests/fixtures/omnidocbench_tiny.json
tests/fixtures/docir_tiny.json
```

Unit tests should cover:

- DocIR serialisation
- stable ID generation
- polygon to bbox conversion
- category mapping
- reading order sorting
- ignore flag handling
- media reference preservation
- Markdown export
- JSONL chunk export
- missing optional dependency errors

## Local regression tests

Use small committed files under:

```text
groundtruth/local/
```

For example:

```text
groundtruth/local/test1.pdf
groundtruth/local/test1.expected.docir.json
```

These tests should validate that the deterministic backend remains stable.

Avoid overly brittle full Markdown snapshots. Prefer structured checks:

- page count
- expected block types
- expected question count
- expected media references
- headers and footers classified as furniture
- stable reading order
- diagrams linked to relevant question blocks where possible

## Optional external benchmarks

Use:

```text
.data/omnidocbench
```

not:

```text
groundtruth/
```

Commands:

```bash
python -m benchmarks.omnidocbench.download
python -m benchmarks.omnidocbench.inspect
python -m benchmarks.omnidocbench.evaluate --backend deterministic --limit 20
python -m benchmarks.omnidocbench.evaluate --backend mineru --limit 20
python -m benchmarks.omnidocbench.evaluate --backend paddleocr_vl --limit 20
python -m benchmarks.omnidocbench.clean
```

## OmniDocBench adapter

Create:

```text
benchmarks/omnidocbench/adapter.py
```

Responsibilities:

- read OmniDocBench JSON
- convert layout_dets to DocIR-like blocks
- preserve category_type
- preserve text
- preserve latex
- preserve html
- preserve poly and bbox
- preserve order
- preserve ignore flag
- preserve attributes
- preserve page_info
- preserve relations

Category mapping:

```text
title -> heading
text_block -> paragraph
table -> table
table_caption -> caption
table_footnote -> footnote
figure -> figure
figure_caption -> caption
figure_footnote -> footnote
equation_isolated -> equation
equation_caption -> caption
header -> header
footer -> footer
page_number -> page_number
page_footnote -> footnote
code_txt -> code
reference -> reference
abandon -> ignored
```

## Benchmark metrics

Internal metrics should start simple and become more sophisticated.

Initial metrics:

- normalised text edit similarity
- block count delta
- reading order similarity
- heading structure similarity
- media reference presence
- formula string similarity
- table HTML similarity placeholder

Later metrics:

- table TEDS
- formula CDM or visual formula comparison
- layout mAP
- caption-link accuracy
- RAG chunk coherence
- retrieval-grounded answer score
- backend disagreement report

## Benchmark reports

Every benchmark run should emit:

```text
results/<run_id>/summary.json
results/<run_id>/pages.jsonl
results/<run_id>/errors.jsonl
results/<run_id>/report.md
```

The report should include:

- backend
- input dataset
- number of pages
- failures
- average metrics
- worst pages
- examples
- raw output paths

## Full official OmniDocBench scoring

Official OmniDocBench evaluation can be added later through Docker or a separate environment.

Do not block internal architecture on official scoring.

The first goal is to use OmniDocBench as:

1. dataset inspiration
2. adapter target
3. optional external evaluation set
4. backend comparison tool

## Synthetic contamination benchmark

Add later:

```text
synthetic article/book generator
rasterised PDFs
blurred pages
JPEG compression
skew
noise
bad text layers
hidden OCR layers
bad font encoding
```

This synthetic benchmark should generate DocIR ground truth first, then LaTeX/PDF/media.

## Acceptance criteria

- default pytest remains fast
- local fixtures validate benchmark code
- OmniDocBench download is optional
- benchmark results are reproducible
- backends can be compared under a single metric framework
- heavy benchmark data can be deleted without affecting the repo
