# Plan 0002: DocIR Core Schema

## Goal

Create a canonical structured representation for documents processed by `doc2md`.

The schema must be robust enough for scientific books, exams, academic papers, reports and scanned documents. It must be AI-readable, RAG-ready and benchmarkable.

## Design principle

Use a Docling-inspired structure, not a simple flat Markdown-like block list.

DocIR should represent both:

1. logical structure
2. physical provenance

Example:

```text
Chapter 3 -> Section 3.2 -> Theorem -> Proof
```

and:

```text
page 128, bbox [x0, y0, x1, y1], extracted by backend X
```

Both are required.


Scientific roles:

```text
chapter_title
section_title
theorem
lemma
corollary
definition
proof
example
exercise
solution
algorithm
bibliographic_item
index_entry
margin_note
```

Exam roles:

```text
difficulty_heading
question
subquestion
answer_option
working_area
```

Use generic `type` plus domain-specific `role`.

Do not explode the base schema with too many specialised types.

## Stable IDs

IDs must be deterministic when possible.

Recommended pattern:

```text
doc_<hash>
page_<index>
block_p<page>_<order>_<short_hash>
group_<role>_<index>
media_page_<index>
```

If bounding boxes are available, they may be part of the ID hash.

Stable IDs are necessary for:

- unit tests
- benchmark comparisons
- RAG traceability
- incremental processing
- caching

## Candidate preservation

BlockIR may contain candidates from multiple backends.

Example:

```json
{
  "block_id": "block_p017_003",
  "type": "question",
  "text": "ABCD is a garden measuring 3 meters by 4 meters...",
  "candidates": [
    {
      "backend": "deterministic_pymupdf",
      "text": "ABCD is a garden measuring 3 meters...",
      "confidence": 0.86
    },
    {
      "backend": "paddleocr_vl",
      "text": "ABCD is a garden measuring 3 metres...",
      "confidence": 0.91
    }
  ]
}
```

## RAG support

DocIR is not the final RAG format. Generate JSONL chunks from it.

Each chunk should include:

- chunk_id
- heading path
- text
- markdown
- page range
- source block IDs
- media references
- semantic role
- provenance
- token estimate
- neighbour chunk IDs

## Benchmark support

Each block should include normalised fields:

```json
"normalised": {
  "text": "...",
  "latex": "...",
  "html": "..."
}
```

Benchmark comparison should operate on DocIR, not on Markdown snapshots.

## Implementation recommendation

Use Pydantic v2 if acceptable as a core dependency. It improves schema validation, serialisation and agent implementation reliability.

If the project wants to avoid new core dependencies, implement dataclasses first and add JSON Schema validation later.

For robustness, Pydantic is recommended.
